The 3 ways ToolbarItem.getToolbarMovementInfo() learns "where the cursor is"

The PR introduces three independent strategies to determine an item's internal navigation state. They're tried in order. Path 1 reaches into private API; Path 3 is a footgun. Source: packages/main/src/ToolbarItem.ts:281–351.

Path 1 — Duck-typed _itemNavigation

If the slotted child has _itemNavigation._getCurrentItem, dig into it.

// ToolbarItem.ts:281–305
_supportsItemNavigationMovementInfo(child) {
  return typeof child._itemNavigation
    ?._getCurrentItem === "function"
    && typeof child._getFocusableItems
       === "function";
}
⚠️ Why fragile

Reaches across the component boundary into private underscore-prefixed members of another component (SegmentedButton, etc.). Renaming _currentItem or restructuring ItemNavigation silently breaks toolbar nav at runtime, with no type-safety net.

Path 2 — Explicit getToolbarMovementInfo

If the child opted in by implementing the interface.

// ToolbarItem.ts:318–320
if (typeof child.getToolbarMovementInfo
    === "function") {
  return child.getToolbarMovementInfo();
}

✓ Implemented on Input & TextArea.
✓ Honest contract — type-checked, opt-in.
✓ This is the one to keep.

Path 3 — Multi-child fallback

If item has >1 slotted child, treat them as a navigable list.

// ToolbarItem.ts:322–351 (paraphrased)
const targets = this._getNavigationTargets();
if (targets.length > 1) {
  const idx = targets.indexOf(active);
  return { currentIndex: idx,
           itemCount: targets.length };
}
⚠️ Why footgun

Slotting any two children — say a <ui5-title> + a <ui5-button> — accidentally turns the item into an arrow-navigable group. The "group" had no semantic meaning intended by the user. Rule of thumb: implicit groupings based on count are landmines.

Proposed simplification — collapse to Path 2 only

Components that currently rely on Path 1 (e.g. SegmentedButton) implement Path 2 in three lines and the duck-typing goes away: 

// SegmentedButton.ts (proposed addition)
getToolbarMovementInfo() {
  const items = this._itemNavigation.getItemsForProcessing();
  const idx   = items.indexOf(this._itemNavigation._currentIndex);
  return idx === -1 ? undefined : { currentIndex: idx, itemCount: items.length };
}

~50 lines deleted in ToolbarItem.ts, plus the fragile private-API dependency. One contract, one mental model.

1
~~Empty input early-exit on Arrow keys~~ — retracted ✓
Input.ts:1664 · TextArea.ts:441 · Toolbar.ts:687
Originally flagged: itemCount <= 1 short-circuit fires on empty inputs. After discussion: this is the desired behavior — inside a toolbar, an empty input has nothing to explore, so arrow keys should move to the next/previous toolbar item.
2
ArrowUp/Down preventDefault is unconditional → breaks <ui5-textarea> caret movement
Toolbar.ts:548 (capture) + Toolbar.ts:663–666
Repro: slot a multi-line <ui5-textarea> in a <ui5-toolbar-item>. The native <textarea> default action is the only source of vertical caret movement (no programmatic Up/Down handler in TextArea). Capture-phase preventDefault on Up/Down kills it before the textarea ever sees the event. ui5-input and ui5-select are unaffected because they handle Up/Down programmatically.

Fix: only preventDefault if the focused item exposes getToolbarMovementInfo(), or move this branch below the movement-info branch.
3
onBeforeRendering reads stale itemsToOverflow, steals focus from overflow button
Toolbar.ts:322–334
Repro: a toolbar with only AlwaysOverflow items, focus on the overflow button. A re-render is triggered. onToolbarItemChange() resets itemsToOverflow = [] (line 559). onBeforeRendering sees the empty array, computes hideOverflowButton === true, and yanks focus to the last navigable item — before prePopulateAlwaysOverflowItems() runs one line later.

Fix: call prePopulateAlwaysOverflowItems() first, or defer the focus-recovery to onAfterRendering after layout has settled.
4
Arrow nav auto-clicks ui5-radio-button, ignoring name grouping
ToolbarItem.ts:199–243 (_handleNavigationTarget)
When focus lands on a [ui5-radio-button], the toolbar calls radio.click(). While that matches the WAI-ARIA radiogroup pattern, it fires for any radio in any grouping — two unrelated yes/no toggles in one toolbar item also get auto-toggled. Wrong altitude: RadioButton-specific behavior baked into a generic toolbar navigator.

Fix: (a) only auto-click when radios share name; or (b) move the behavior into RadioButton itself (let it react to focus); or (c) introduce a small protocol like child.onToolbarArrowFocus?.().
5
_focusOutsideToolbar walks the entire document on every Tab keypress
Toolbar.ts:861
getTabbableElements(document.body) recursively descends every shadow root and runs isElementTabbable (which calls getComputedStyle) on each. On a typical Fiori shell page this is a synchronous O(N) walk per Tab, with potential layout flushes.

Deeper question: if roving tabindex is correct, the browser already does this for free — that's literally what Tab does. The handler may be solving a problem that doesn't exist.

Fix: scope to this.getRootNode(), or delete the handler and rely on native Tab.
6
handlesOwnKeyboardNavigation is dead public API
ToolbarItemBase.ts:76–78 · ToolbarItem.ts:69–78
grep -rn handlesOwnKeyboardNavigation packages/*/src returns only the two getter definitions — nothing reads them. Earlier branch checkpoints had four call-sites; they were removed during refactor and the getters were left behind.

Fix: remove the getter (preferred) — getToolbarMovementInfo() truthiness already serves the same purpose.
7
RadioButton @csspart root JSDoc removed but template still emits part="root"
RadioButton.ts:69–70 (deleted) · RadioButtonTemplate.tsx:10
Commit 7622f8045cc deleted the @csspart root JSDoc but left part="root" on the template. Either revert the doc deletion (the part is still public) or also drop the part from the template (silently breaks ::part(root) consumers).
Looks accidental and unrelated to the toolbar feature.

Duplicated getToolbarMovementInfo in Input/TextArea + duplicated currentIndex discovery in ToolbarItem
Input.ts:1664–1683 · TextArea.ts:441–460 · ToolbarItem.ts (3× variants)
Input/TextArea bodies are mechanically identical: selectionStart, value.length, focus check, +1. Extract to caretMovementInfo(el) next to Caret.ts.

9
Selection ignores selectionEnd
Input.ts:1676 · TextArea.ts:453
Reports currentIndex = selectionStart regardless of direction. With text fully selected (start=0, end=N), ArrowLeft hits atBackwardBoundary (currentIndex <= 0) and exits the input — but native behavior would collapse the selection to position 0, still inside the field.

Fix: use selectionEnd for forward comparisons, selectionStart for backward.

What the mock simulates

  • Roving tabindex: exactly one item has tabindex=0; the rest have tabindex=-1. Tab-out works because only one tab stop.
  • ArrowLeft/Right: moves focus between items. Inputs/textarea use selectionStart + value.length to decide when to exit.
  • Empty input: first arrow press exits (toolbar takes over — Finding #1 retracted).
  • "hello world" input: arrow moves caret first, only exits at boundaries (proper movementInfo behavior).
  • Textarea ArrowDown: in the real PR, capture-phase preventDefault kills the native vertical caret. The mock highlights this case in red in the log.
  • Home/End: jump to first/last toolbar item.
