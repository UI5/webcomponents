Toolbar — bug reproductions for PR #13622

Three reproducible bugs in the WAI-ARIA toolbar keyboard navigation PR. Open this page, follow the steps in each section, and the live "Observed" log shows what actually fires.

CRITBug 1 — Empty ui5-input exits on first ArrowLeft / ArrowRight

Toolbar.ts:686-712 + Input.ts:1664-1683. With an empty value, itemCount = valueLength + 1 = 1; the boundary check short-circuits on itemCount <= 1, treating the caret as being at both start and end simultaneously, so the very first arrow press exits the input.

  1. Click into the empty input above.
  2. Press ArrowRight.
Expected: caret stays in the input (nothing to traverse).
Actual: focus jumps to the After button. Symmetric for ArrowLeft → Before.

Try with text typed in: caret moves normally and only exits at the boundaries — that part works. The bug is specifically the empty-string case.

CRITBug 2 — ArrowUp / ArrowDown can't move caret across textarea lines

Toolbar.ts:548, 663-666. The toolbar attaches its keydown listener with capture: true and unconditionally preventDefault()s on Up/Down. Native <textarea> caret movement is the default action — once it's prevented, the caret cannot leave the first line.

  1. Click into the textarea, place the caret anywhere on line one.
  2. Press ArrowDown.
Expected: caret moves to line two.
Actual: caret stays put. The log shows defaultPrevented=true on the ArrowDown event.

Compare with a plain <textarea> outside any toolbar — same key works fine. The toolbar's e.preventDefault() in capture phase is the only difference.

HIGHBug 4 — Arrow nav auto-selects radio buttons regardless of group

ToolbarItem.ts:199-243. _handleNavigationTarget calls radio.click() on any element with the [ui5-radio-button] attribute. It ignores the name attribute (which defines the actual group), so arrow-navigating between unrelated radios silently mutates form state.

  1. Tab into the first radio (Option A) — don't click. Just have it focused.
  2. Press ArrowRight.
Expected: focus moves to Option B; neither radio is checked (they aren't in the same group; arrow nav should not select).
Actual: focus moves AND Option B becomes checked (change event fires). Press ArrowLeft → Option A becomes checked.

The two radios have different name values (group-A vs group-B), so they should not behave as a group. The PR's marker-attribute check ignores this.
