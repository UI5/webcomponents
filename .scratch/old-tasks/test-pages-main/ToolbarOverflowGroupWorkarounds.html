Toolbar — can we live without overflowGroup?

Each scenario states what a Fiori app developer wants, what they'd see if the toolbar "did the right thing," and what actually happens with today's APIs. Drag the right edge of the dashed box to shrink the toolbar and watch the overflow.
Tip: open the overflow popover with the ··· button to inspect popover layout. The markup block above each toolbar shows the exact HTML used in the demo.

Scenario 1 — Search label must never separate from its input

A toolbar has a few actions and one "Search: [____]" pair. If the bar gets narrow, the label and the input must move to the overflow popover together — never one without the other.

Workaround: wrap the label and input in a single <ui5-toolbar-item> so the toolbar measures them as one item.

<ui5-toolbar>
  <ui5-toolbar-button icon="add"      text="Action A"></ui5-toolbar-button>
  <ui5-toolbar-button icon="employee" text="Action B"></ui5-toolbar-button>
  <ui5-toolbar-separator></ui5-toolbar-separator>

  <!-- the workaround: one wrapper, both children inside -->
  <ui5-toolbar-item class="has-inline-children">
    <span class="search-pair">
      <ui5-label for="s1" show-colon>Search</ui5-label>
      <ui5-input id="s1" placeholder="Type…"></ui5-input>
    </span>
  </ui5-toolbar-item>

  <ui5-toolbar-button icon="decline" text="Action C"></ui5-toolbar-button>
</ui5-toolbar>

Expected: as the bar shrinks, "Search:" + input either both stay visible or both move to overflow.

Actual: the wrapper crosses the boundary atomically. Resize down — at the narrow end both are in the popover, the rest of the bar shows "Action A / Action B" only.

Caveat noted in the footer: the wrapper needs an explicit display on the host to be measured at all.

Scenario 2 — Two related buttons must travel together

After the separator, "Sort" and "Filter" are a UX pair. They are still two independent buttons (each focusable, each with its own tooltip), but they should never split across the bar / popover boundary.

Workaround: put both buttons inside one <ui5-toolbar-item>.

<ui5-toolbar>
  <ui5-toolbar-button text="Primary 1"></ui5-toolbar-button>
  <ui5-toolbar-button text="Primary 2"></ui5-toolbar-button>
  <ui5-toolbar-button text="Primary 3"></ui5-toolbar-button>
  <ui5-toolbar-separator></ui5-toolbar-separator>

  <!-- the workaround: two raw ui5-button inside one wrapper -->
  <ui5-toolbar-item class="has-inline-children">
    <span class="cluster">
      <ui5-button icon="sort">Sort</ui5-button>
      <ui5-button icon="filter">Filter</ui5-button>
    </span>
  </ui5-toolbar-item>
</ui5-toolbar>

Expected: Sort + Filter overflow together; in the popover they appear as two rows (one per button), each individually focusable with arrow keys, each individually labeled in the a11y tree.

Actual: they DO overflow together — but in the popover they render as one horizontal row, not stacked. Open the popover, then try arrow-key navigation: ↓ skips past both at once because the popover sees one item (the wrapper), not two children.

Scenario 3 — A cluster of "secondary" actions should always be in overflow

Three secondary actions ("Secondary 1/2/3") should never compete for bar space with the primary actions. They live in the overflow popover at all widths.

Workaround: set overflow-priority="AlwaysOverflow" on each of them.

<ui5-toolbar>
  <ui5-toolbar-button icon="add" text="Primary 1"></ui5-toolbar-button>
  <ui5-toolbar-button icon="add" text="Primary 2"></ui5-toolbar-button>

  <!-- the workaround: mark each secondary AlwaysOverflow -->
  <ui5-toolbar-button text="Secondary 1" overflow-priority="AlwaysOverflow"></ui5-toolbar-button>
  <ui5-toolbar-button text="Secondary 2" overflow-priority="AlwaysOverflow"></ui5-toolbar-button>
  <ui5-toolbar-button text="Secondary 3" overflow-priority="AlwaysOverflow"></ui5-toolbar-button>
</ui5-toolbar>

Expected: they appear only in the popover, regardless of bar width. The ··· button is always visible.

Actual: works as expected. But notice: this is "always in overflow," not "in overflow only when needed." There's no knob for "keep them in the bar when there's room, push them all out together when there isn't."

Scenario 4 — A cluster of "must-keep" actions should never overflow

"Stay 1" and "Stay 2" are critical actions. They should stay visible in the bar even when other items have to move to overflow.

Workaround: set overflow-priority="NeverOverflow" on each.

<ui5-toolbar>
  <!-- the workaround: mark each "stay" NeverOverflow -->
  <ui5-toolbar-button text="Stay 1" overflow-priority="NeverOverflow"></ui5-toolbar-button>
  <ui5-toolbar-button text="Stay 2" overflow-priority="NeverOverflow"></ui5-toolbar-button>
  <ui5-toolbar-separator></ui5-toolbar-separator>
  <ui5-toolbar-button icon="add" text="Move 1"></ui5-toolbar-button>
  <ui5-toolbar-button icon="add" text="Move 2"></ui5-toolbar-button>
  <ui5-toolbar-button icon="add" text="Move 3"></ui5-toolbar-button>
</ui5-toolbar>

Expected: "Stay 1/2" stay visible. The other ("Move 1/2/3") move to the popover first as the bar narrows.

Actual: works as expected. Edge case: if even the NeverOverflow items can't fit, the toolbar runs out of room — that's the only failure mode, and it's unrelated to grouping.

Scenario 5 — Tier-based importance: each tier should travel as a unit

Items belong to three importance tiers. Lower tiers should move to overflow first. Within each tier, the items should overflow together — Tier 3 leaves as a pair, then Tier 2 leaves as a pair, etc.

Workaround: order items by importance in the DOM (last in DOM = first to overflow). Today there is no grouping API.

<ui5-toolbar>
  <!-- DOM order = priority. Last in DOM overflows first.
       BUT: no way to say "Tier 3 must overflow as a unit". -->
  <ui5-toolbar-button text="Tier 1 - A"></ui5-toolbar-button>
  <ui5-toolbar-button text="Tier 1 - B"></ui5-toolbar-button>
  <ui5-toolbar-separator></ui5-toolbar-separator>
  <ui5-toolbar-button text="Tier 2 - A"></ui5-toolbar-button>
  <ui5-toolbar-button text="Tier 2 - B"></ui5-toolbar-button>
  <ui5-toolbar-separator></ui5-toolbar-separator>
  <ui5-toolbar-button text="Tier 3 - A"></ui5-toolbar-button>
  <ui5-toolbar-button text="Tier 3 - B"></ui5-toolbar-button>
</ui5-toolbar>

Expected: as you shrink the bar, all of Tier 3 disappears at once, then all of Tier 2.

Actual: items leave one at a time, breaking tier atomicity. Empirical measurements from this page:
at 480 px — only "Tier 3 - B" is in overflow (Tier 3 - A still in bar);
at 400 px — both Tier 3 items in overflow;
at 360 px — Tier 3 plus "Tier 2 - B" in overflow, but Tier 2 - A still in bar.
The tier is split at two distinct breakpoints. No combination of existing properties prevents this.

This is the case overflowGroup would solve cleanly: mark all Tier 3 items overflow-group="3", all Tier 2 overflow-group="2", etc. Whether the workaround above is "good enough" depends on whether any real Fiori app actually files this complaint.

Scenario 6 — The hard case: 3 independent buttons must travel together AND remain individually focusable in overflow

"View, Sort, Filter" are a UX cluster. They must overflow as a unit. In the popover, each must remain its own focus stop with its own a11y label — they cannot collapse into one wrapper.

Workaround attempt: wrap them in one <ui5-toolbar-item>, as in Scenario 2.

<ui5-toolbar>
  <ui5-toolbar-button icon="add"      text="Create"></ui5-toolbar-button>
  <ui5-toolbar-button icon="employee" text="Assign"></ui5-toolbar-button>
  <ui5-toolbar-separator></ui5-toolbar-separator>

  <!-- three independent buttons inside one wrapper -->
  <ui5-toolbar-item class="has-inline-children">
    <span class="cluster">
      <ui5-button icon="grid">View</ui5-button>
      <ui5-button icon="sort">Sort</ui5-button>
      <ui5-button icon="filter">Filter</ui5-button>
    </span>
  </ui5-toolbar-item>
</ui5-toolbar>

Expected: in the popover, three stacked rows (View / Sort / Filter), three arrow-key stops, three distinct accessible names.

Actual: one horizontal row in the popover (open it and look), one arrow-key stop, one accessible item in the a11y tree. The cluster travels together — but loses per-button structure on the way. This is the failure mode the wrap workaround can't escape.

Summary

Three scenarios (1, 3, 4) are covered by today's APIs. Two (5, 6) are not. Scenario 2 is partly covered — the items travel together but lose per-item structure in the popover.

The case for overflowGroup hinges on whether Scenarios 5 and 6 are real customer needs or hypotheticals. With no concrete report against the current API, the right move is to document the workarounds, keep overflowGroup on hold, and revisit when a real complaint lands.

Finding A — separate bug in ui5-toolbar-item

While building this page I found that <ui5-toolbar-item> with arbitrary slotted content (not a known UI5 component) measures as 0 px wide inside the toolbar's overflow math. Its CSS is just :host { height: 100% } with no display rule, so the host's offsetWidth stays 0 regardless of slotted content. The fix is to add display: inline-flex (or similar) on the host. Every scenario on this page that uses a wrapper sets that explicitly via the .has-inline-children class. This is worth filing independently of any overflowGroup decision.
